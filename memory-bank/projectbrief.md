# Project Brief: Zen Reader

## Обзор проекта
Zen Reader — веб-приложение для чтения (предположительно PDF, на основе зависимостей), разрабатываемое с фокусом на качественный код и строгую документацию.

## Правила пользователя (User Rules)

### Глобальные правила
1.  **Язык**: Всегда отвечай по-русски.
2.  **Тестирование**: Всегда при тестировании сайтов и поиске материалов в интернете используй **BrowserMCP**.
3.  **Терминал**: Все команды в терминале (git, сборка, тесты, установка пакетов) пользователь выполняет **ВРУЧНУЮ**.
    *   Ты **НЕ** должен запускать их автоматически.
    *   Предоставляй готовую команду в блоке кода.
4.  **Роль**: Помощник в разработке ПО (роль: Cursor).

### Работа с контекстом (Memory Bank)
Перед началом любой работы **ОБЯЗАТЕЛЬНО**:
1.  Прочитай файлы Memory Bank в порядке:
    *   `projectbrief.md`
    *   `productContext.md`
    *   `systemPatterns.md`
    *   `techContext.md`
    *   `activeContext.md`
    *   `progress.md`
2.  В ответе укажи дату и дай однострочное резюме для каждого файла.
3.  Если файла нет — создай шаблон.

### Документирование и Форматирование
1.  **Changelog** (`/docs/changelog.md`):
    *   Формат: `## [YYYY-MM-DD] - Краткое описание`
    *   Секции: Добавлено / Изменено / Исправлено.
2.  **Task Tracker** (`/docs/tasktracker.md`):
    *   Шаблон задачи с полями: Статус, Описание, Шаги выполнения, Зависимости.
    *   Требовать подтверждение перед каждым новым шагом.
3.  **Код файлов**:
    *   Header при создании: `/** @file: ... @description: ... @dependencies: ... @created: ... */`
4.  **Project Update**: Обновлять `/docs/project.md` после релизов (с Mermaid).

### Процесс разработки
1.  **Принципы**: SOLID, KISS, DRY.
2.  **Code Review**: Результаты в `changelog.md` и `project.md`.
3.  **Тестирование**: Обязательное функциональное тестирование (BrowserMCP/Testsprite) после крупных задач.
4.  **Стиль**: Единый стиль кодирования.
5.  **Чистота**: Удалять неиспользуемый код перед мержем.
6.  **Planner Mode**: Задать 4–6 вопросов перед планом.

### Политика Git
1.  **Процесс**: Обнови Memory Bank/Docs -> Команда `git add/commit` -> Команда `git push`.
2.  **Частота**: Мелкие и частые коммиты (1-3 пуша в день).
3.  **Сообщения**: Conventional Commits (`feat:`, `fix:`, `chore:`, etc.).
4.  **Ветки**: `feature/*`, `fix/*`, `chore/*` -> PR -> `main`.
5.  **Безопасность**: Не коммитить мусор (`.gitignore`).
